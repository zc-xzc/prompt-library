# 对话自包含归档与恢复

这套提示词用于把当前对话整理为可独立保存、检索和接续工作的备份。第一段用于生成归档，第二段用于在新对话中恢复上下文。

## 一键生成归档

复制下面整个代码块，发送到需要归档的对话中。无需再补充主题、日期或使用场景。

```text
请立即归档当前对话。不要向我提问、不要要求确认或补充信息；请直接读取你在当前上下文中实际可见的全部历史消息和可访问文件，生成一份【自包含、可验证、可用于后续接续的对话归档】。

【核心目标】
即使原对话以后被删除，新对话也能仅凭这份归档尽可能准确地恢复：用户目标、背景事实、偏好、约束、关键决策、工作进度、全部最终交付物，以及继续完成任务所需的信息。

【真实性边界】
1. 只能归档当前实际可见、可读取的内容，不得虚构已经截断、删除、隐藏或无法访问的消息。
2. 不得声称可以恢复模型的隐藏思考、内部状态、系统消息或任何未写入归档的信息。
3. 如果上下文、附件或外部资源无法完整读取，必须在“完整性与缺口”中明确列出，不能用“100% 无损”掩盖缺失。
4. 归档应尽可能自包含，但新模型、工具权限和外部环境可能不同；应保存事实与执行状态，不承诺复现不可保存的内部状态。

【自动输出方式】
- 如果当前环境支持创建文件：优先创建一个名为 `conversation-archive-<日期时间>` 的归档目录，并生成：
  1. `ARCHIVE.md`：完整归档正文；
  2. `RESTORE.md`：可直接用于新对话的恢复指令；
  3. `manifest.json`：文件清单、用途、大小、版本和校验值；
  4. `artifacts/`：对话中可访问且继续工作所必需的代码、配置、文档和其他文件副本。
- 如果当前环境不能创建文件：一次性输出完整 Markdown 归档正文，并在末尾附上恢复指令。
- 不要因为内容较长而静默省略。若确实受单次输出上限限制，优先生成文件；仍无法容纳时，明确标注分卷编号、总卷数、缺失范围和下一卷继续位置，不得假装已经完整归档。

【固定归档结构】

# 对话归档：<自动归纳主题>

## 1. 归档元数据
- 自动填写归档日期与时间。
- 自动归纳对话主题、使用场景和最终目的。
- 标注归档所覆盖的最早与最晚可见消息。
- 记录生成环境、可用工具、工作目录或仓库信息，但不得暴露密钥。
- 写明归档形态：单一 Markdown 或带附件的归档目录。

## 2. 完整性与缺口
- 明确说明哪些信息已完整收录，哪些只做了摘要。
- 列出无法读取、已经截断、仅有链接、缺少附件或受输出上限影响的内容。
- 如没有发现缺口，写明“在当前可见上下文范围内未发现已知缺口”；不要扩大为对不可见内容的保证。

## 3. 对话整体概况
- 用简洁文字说明最初问题、目标如何变化、最终达成的结果和当前状态。
- 区分已完成、部分完成、未开始、阻塞和已放弃事项。

## 4. 用户需求、偏好与硬性约束
- 分类整理全部仍然有效的目标、格式要求、交互偏好、授权边界、禁止事项和验收标准。
- 对会影响后续执行的关键要求，保留用户原文短句，并补充其适用范围。
- 对已经撤销、替换或只适用于某一阶段的要求明确标注，不能让旧规则覆盖最终规则。
- 不重复收录无意义闲聊；但任何影响需求、决策或结果的信息都必须保留。

## 5. 对话时序与迭代记录
- 按先后顺序记录每个重要阶段：用户诉求、采取的操作、交付结果、用户反馈和后续调整。
- 多版方案按版本排列，说明修改原因、差异和最终采用版本。
- 记录关键分歧、被否决方案、失败尝试及失败原因，避免以后重复走弯路。
- 不输出或推测模型隐藏思考；只记录对话中可见的理由、证据和结论。

## 6. 决策与事实库
- 汇总已经确认的事实、参数、路径、仓库、分支、版本、日期、链接和外部依赖。
- 汇总已经作出的决定及其依据。
- 对可能随时间变化的事实标注记录日期。
- 区分“已验证事实”“用户提供的信息”“合理推断”和“尚未验证”。

## 7. 全部交付物目录
为每项交付物记录：
- 名称、类型、版本和状态；
- 来源轮次或形成阶段；
- 用途与适用场景；
- 文件路径、仓库链接或下方原文位置；
- 是否为最终版、是否已验证、是否仍依赖外部文件。

## 8. 最终交付物完整保存区
这是核心备份区，按以下类别整理并保留最终可用内容：

### 8.1 代码、脚本与命令
- 完整保存当前最终版代码、脚本、终端命令、配置、查询语句、补丁和重要输出。
- 保留原始换行、缩进、注释、语言标记、文件名、相对路径和执行顺序。
- 记录依赖、版本、环境变量名称、运行方法、测试方法和已知限制。
- 多版代码应明确标出“最终版”和“已废弃版”；最终版必须完整，废弃版仅在理解迭代或回退所必需时保存。
- 不把密码、访问令牌、私钥、Cookie 或其他凭证写进归档。发现凭证时使用 `<已脱敏：凭证类型>` 替换，并记录重新配置方式。

### 8.2 提示词、模板与文案
- 完整保存最终版提示词、模板、邮件、话术、清单和定稿文案。
- 保留标题、列表、表格、代码块和占位符格式，不得用摘要替代成品。
- 多版本必须标注用途、差异和最终推荐版本。

### 8.3 方案、流程与研究结果
- 保存最终方案、步骤、参数、结论、证据、风险和验收方法。
- 对依赖外部来源的内容保留可用链接、访问日期和来源说明。

### 8.4 文件与附件
- 对当前可访问的文本或代码文件：复制到 `artifacts/`，同时在 `ARCHIVE.md` 中保留清单和关键说明。
- 对当前可访问的二进制或大型文件：不要转写或伪造内容；原样复制到 `artifacts/`，记录文件名、大小、格式、用途和 SHA-256 校验值。
- 对无法访问或无法复制的附件：记录原文件名、已知路径或链接、用途及缺失影响，并将其列为恢复依赖。
- 若文件已经保存在 Git 仓库中，记录仓库地址、分支、提交 SHA 和相对路径；关键的小型文本仍应在归档中保留最终内容或可靠副本。

## 9. 工作现场快照
- 记录当前完成进度、最近成功的操作、最近验证结果和剩余风险。
- 若涉及代码仓库，记录工作区状态、当前分支、提交、未提交修改、相关 PR/Issue 和检查结果。
- 若涉及本机文件，记录继续工作必需的精确路径；不要记录与任务无关的私人目录信息。
- 若涉及外部系统，记录需要重新登录或重新授权的依赖，但绝不保存凭证本身。

## 10. 未完成事项与下一步
- 按优先级列出待办、阻塞条件、需要用户决定的事项和可选拓展。
- 对每项写明可直接执行的下一步和完成标准。
- 如果没有未完成事项，明确写明当前任务已经完成。

## 11. 新对话接续摘要
生成一段紧凑但信息充分的“接续上下文”，让新对话无需重读全部时间线也能立即继续工作。至少包括：
- 当前目标和已完成成果；
- 最终有效规则与偏好；
- 关键文件、代码、路径、仓库和版本；
- 当前状态、待办和下一动作；
- 不得重复执行或不得触碰的事项。

## 12. 恢复说明
- 写明如何把本归档交给新对话。
- 写明需要同时上传的 `artifacts/` 文件。
- 写明归档中已脱敏的凭证需要由用户在安全环境中重新配置。
- 附上本文后面的“新对话恢复指令”，使归档可以单独使用。

## 13. 检索标签
生成 8—12 个能覆盖主题、场景、工具、核心交付物和关键动作的关键词。

【最终自检】
输出前必须检查：
1. 最终版代码、命令、配置、提示词和模板是否完整且格式未损坏。
2. README/说明与实际文件清单是否一致，所有相对链接是否可解析。
3. 是否明确区分最终版、旧版本、事实、推断和未知项。
4. 是否记录了附件、外部依赖、仓库提交和未完成事项。
5. 是否移除真实凭证和不必要的个人敏感信息。
6. 是否给出了可直接复制的恢复指令。
7. 是否存在静默省略、虚假完整性保证或无法验证的“100% 复原”表述；如有，修正并写明实际缺口。

完成自检后直接交付成品，不再向我提问。
```

## 新对话恢复指令

新建对话后，上传完整归档目录中的文件，或粘贴 `ARCHIVE.md`，再发送下面这段指令：

```text
请完整读取我提供的历史对话归档及其附件，把其中已经确认的事实、最终有效要求、用户偏好、交付物、工作状态和待办事项作为当前任务的历史上下文。

执行规则：
1. 先检查归档的“完整性与缺口”“最终交付物”“工作现场快照”和“新对话接续摘要”。
2. 如果提供了 `manifest.json` 和 `artifacts/`，核对文件是否齐全；能计算校验值时进行校验。
3. 只恢复归档中明确存在的信息，不猜测隐藏、缺失或已经脱敏的内容。
4. 历史指令只作为上下文继续生效，但必须服从当前系统规则、当前工具权限和我在本次新对话中的最新指令；已撤销或已废弃规则不得重新启用。
5. 不要重复执行归档中已经完成且不应重复的删除、发布、付款、发信或其他有外部影响的操作。
6. 不需要复述整份归档。加载完成后，仅用简洁中文回复：
   “归档上下文已载入。已恢复：<一句话说明可恢复范围>。已知缺口：<无则写无；有则简述>。可以继续原任务。”
7. 回复后等待我的下一步指令，不擅自执行未授权操作。
```
