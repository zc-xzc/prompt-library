# 使用方法、能力边界与隐私注意事项

## 它能保存什么

这份提示词会尽可能保存当前对话中可见的需求、规则、决策、代码、命令、模板、文件清单、验证结果和下一步。若环境支持文件操作，还会要求建立带清单和校验值的归档目录。

它适合长任务收尾、准备删除旧对话、跨模型继续工作、保存技术调试过程，以及把重要对话转成长期知识资产。

## 为什么不承诺“100% 复原”

任何纯文本归档都无法保存模型的隐藏内部状态，也无法读取已经截断、删除或当前无权访问的消息。没有一起保存的附件、网页和外部系统状态同样可能变化。因此，可靠做法是明确覆盖范围与缺口，并完整保存继续工作真正需要的成果和现场状态。

在可见上下文、最终交付物和必要附件都完整的前提下，这份归档可以高度还原任务上下文并支持继续工作；“可验证的完整”比无法核实的“100% 无损”更可靠。

## 代码与文件如何处理

- 小型代码、配置、脚本和命令应以原格式写入 `ARCHIVE.md`。
- 可访问的完整文件应复制到 `artifacts/`，并在 `manifest.json` 中记录 SHA-256。
- Git 仓库应记录远端地址、分支、提交 SHA 和相关 PR，而不是只保存一句“已经上传”。
- 大型或二进制附件必须原样保存；仅有文字摘要不能替代原文件。
- 如果附件不可访问，归档必须明确记录缺失项及其影响。

## 隐私与安全

本仓库是公开仓库，只保存通用提示词。不要把生成的私人对话归档、客户资料、聊天记录、内部代码或凭证提交到这里。

归档中不应保存密码、访问令牌、私钥、Cookie 和恢复码。使用脱敏占位符，并在新环境中通过安全方式重新配置。

## 推荐使用方式

1. 在准备结束或删除的重要对话中发送 [归档生成提示词](prompt.md#一键生成归档)。
2. 下载并保存生成的完整目录，确认 `ARCHIVE.md`、`RESTORE.md`、`manifest.json` 和 `artifacts/` 齐全。
3. 至少抽查最终代码、关键模板、未完成事项和附件校验值。
4. 将归档保存在受控的本地目录或加密备份中。
5. 新开对话时上传归档，并使用 [恢复指令](prompt.md#新对话恢复指令)。

## 什么时候使用精简版

如果只是保存一次短问答，可以把归档生成提示词中的固定结构缩减为：概况、关键需求、最终答案、待办和恢复摘要。涉及代码、文件、长时间迭代或重要决策时，建议使用完整版。

## 真实使用案例

### Codex 对话数据恢复 (2026-07-21)

使用本 prompt 对一次 Codex Desktop 对话恢复任务进行了归档。该对话涉及：
- 诊断并恢复 29 个丢失的 Codex 历史对话
- 合并两个 state_5.sqlite 数据库（根目录 + sqlite/ 子目录）
- C 盘到 D 盘数据迁移与冗余清理

完整归档见 examples/codex-recovery-20260721/，包含 ARCHIVE.md、RESTORE.md、manifest.json 和 artifacts/。
